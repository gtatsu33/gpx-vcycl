1. 技術スタック
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  ホスティング   : Cloudflare Workers（静的アセット配信 + Workerスクリプト。
                   @cloudflare/vite-plugin 経由でVite/Workers runtimeを統合）
  フロントエンド : Vanilla JavaScript（ESモジュール、フレームワークなし）
  ビルド        : Vite 8 + @cloudflare/vite-plugin
  ストレージ     : IndexedDB（idbライブラリでラップ）
  BLE           : Web Bluetooth API（navigator.bluetooth）
  地図(2D)       : Leaflet.js（OpenStreetMapタイル）
  地図(2D、代替) : Google Maps JavaScript API + Street View
                   （オーナーモード限定、googleMapView.js）
  3Dマップ       : Three.js（WebGLRenderer、手組みのジオメトリ生成）
  外部DB(リモートGPX): Supabase JS クライアント（Storage読み取りのみ）
  サーバー処理    : Cloudflare Workers（単一Workerスクリプト src/worker/index.js。
                   Strava OAuth秘匿・オーナーモード認証・Street View中継のみ。
                   それ以外はクライアント完結）
  デプロイCLI     : wrangler 4
  テスト         : Vitest + jsdom
  CSSフレームワーク: なし（素のCSS、index.html内に直書き）

  ※ npm run build は @cloudflare/vite-plugin により2つのVite environment
    （Worker用・クライアント用）をビルドし、dist/ 以下に出力する。
    npm run deploy で vite build → wrangler deploy を実行し、
    Cloudflare Workersへ静的アセット＋Workerスクリプトを一体でアップロードする。

4. 環境変数
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Vite はクライアントに公開する環境変数に VITE_ プレフィックスが必要。

  変数名                       用途                              参照箇所
  ──────────────────────────   ────────────────────────────────  ─────────────────
  VITE_MAPILLARY_TOKEN         実写風景写真APIトークン            main.js, mapillary/api.js
                                （未設定で機能自体を無効化）
  VITE_SUPABASE_URL            リモートGPXソース(Supabase)URL     storage/remoteRoutes.js
  VITE_SUPABASE_ANON_KEY       同publishable key                 storage/remoteRoutes.js
                                （service_role key は使わない。db.v4.md参照）
  VITE_SUPABASE_BUCKET         同バケット名（既定 gpx_routes）    storage/remoteRoutes.js
  VITE_STRAVA_CLIENT_ID        Strava OAuth クライアントID        strava/auth.js
  VITE_STRAVA_REDIRECT_URI     Strava OAuth リダイレクトURI       strava/auth.js
                                （本番URLと開発URLで値が異なるため
                                 .env.production に本番用の値を置く。後述）
  VITE_GOOGLE_MAPS_API_KEY     Google Maps/Street View（オーナー） ui/googleMapView.js
  VITE_GOOGLE_MAPS_MAP_ID      Google Maps Map ID                 ui/googleMapView.js
                                （AdvancedMarkerElement使用に必須）

  サーバー側専用（Cloudflare Workers のシークレット/環境変数。
  VITE_プレフィックス無し、クライアントバンドルに含まれない）:
    STRAVA_CLIENT_ID            src/worker/index.js が使用
                                 （VITE_STRAVA_CLIENT_ID と同じ値を別途設定する）
    STRAVA_CLIENT_SECRET        src/worker/index.js が使用
    OWNER_PASSCODE               src/worker/index.js が使用
    GOOGLE_MAPS_STREETVIEW_KEY   src/worker/index.js が使用
                                 （/api/streetview 経由でのみクライアントに渡る。
                                 VITE_GOOGLE_MAPS_API_KEY とは別キーでよい）

  設定方法（15章「Cloudflare Workers デプロイ設定」参照）:
    - VITE_* はビルド時変数。.env（共通）/ .env.production（本番URL等の
      上書き）/ .env.local（個人のローカル秘密値、git管理外）に配置する
    - サーバー側専用の4つは実行時シークレット。ローカルでは .dev.vars
      （.dev.vars.example をコピーして作成、git管理外）、本番は
      `wrangler secret put <NAME>` または `wrangler deploy
      --secrets-file .dev.vars` で設定する（13章参照）

5. IndexedDBスキーマ（src/storage/db.js）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  DB名: virtualCycling　　DBバージョン: 3（idbのopenDB、upgrade内で
  バージョンごとに段階的にストアを追加）

  ストア          keyPath        追加バージョン   内容
  ──────────────  ─────────────  ──────────────  ─────────────────────────
  routes          id (auto)      v1               保存済みGPXルート
  rides           id (auto)      v1               保存済みライド記録
  settings        なし(KV)       v1               設定値・トークン類
  workouts        id (auto)      v2               保存済みZWOワークアウト
  mapillaryCache  なし(KV)       v3               実写写真選定結果キャッシュ

  ※ インデックス（createIndex）は使用していない。一覧はgetAll()で
    全件取得し、必要に応じてJS側でソート・フィルタする。

  エクスポート:
    initDb(): Promise<IDBPDatabase>   起動時に1回呼ぶ。DBを開き
      アップグレードし、モジュール内キャッシュに保持する。
    getDb(): IDBPDatabase             キャッシュ済みDBを返す。
      initDb()未呼び出し時は例外を投げる。

5-1. routes ストア（レコード形状）
────────────────────────────────────────
  { id, name, gpxText, totalDistanceM, totalElevationGainM, savedAt }

  API（storage/routes.js）:
    saveRoute({name, gpxText}) → id     Route.fromGpx()で検証してから保存
    getRoute(id) → record|undefined
    listRoutes() → record[]
    deleteRoute(id) → void

5-2. rides ストア（レコード形状）
────────────────────────────────────────
  {
    id, routeId, routeName, startedAt, endedAt,
    totalDistanceM, totalElapsedSec,
    avgPowerW, avgCadenceRpm, avgHeartRateBpm,
    samples: [{ timestampMs, lat?, lon?, elevationM?, distanceM,
                velocityMs, gradientPercent?, powerW, cadenceRpm,
                heartRateBpm }],
    uploadStatus: 'pending' | 'uploaded' | 'failed',
    uploadedAt, stravaActivityId, uploadError,
  }
  （lat/lon/elevationM/gradientPercentはルートモードのみ。ワークアウト
    記録には含まれない）

  API（storage/rides.js）:
    saveRide(summary) → id              集計値を計算して保存
    listRides() → record[]              startedAt降順でソート

    markUploaded(rideId, stravaActivityId) → key
    markUploadFailed(rideId, errorMsg) → key
    deleteRide(id) → void

5-5. remoteRoutes / remoteWorkouts（Supabase Storage、IndexedDB外）
────────────────────────────────────────
  storage/remoteRoutes.js:
    listRemoteGpxFiles() → [{name, id, updated_at}]  バケット直下を
      name昇順・上限200件でlist()し、.gpx拡張子のみフィルタ
    fetchRouteFilesMeta() → Map<file_key, {displayName, distanceM,
      elevationGainM}>  route_filesテーブルをSELECTし、file_key を
      キーにしたMapへ変換（db.v4.md「読み取りフロー（gpx-vcycl）」の
      設計をそのまま実装）。routePicker.js がこれと
      listRemoteGpxFiles()の結果をfile_key(=file.name)で突き合わせ、
      displayNameがあればそれを、無ければfile_key（拡張子除去）を
      表示名として使う
    downloadRemoteGpx(fileName) → gpxText文字列

  storage/remoteWorkouts.js（remoteRoutes.jsと対称の実装。GPXと同じ
    gpx_routesバケットを共用し、.zwo拡張子でフィルタする点のみ異なる）:
    listRemoteZwoFiles() → [{name, id, updated_at}]  .zwo拡張子のみフィルタ
    fetchWorkoutFilesMeta() → Map<file_key, {displayName, durationS}>
      workout_filesテーブルをSELECT（db.v4.md「読み取りフロー
      （gpx-vcycl、ワークアウト）」参照）
    downloadRemoteZwo(fileName) → zwoText文字列

────────────────────────────────────────
6-1. constants.js
────────────────────────────────────────
  FTMS: FTMS_SERVICE_UUID(0x1826), INDOOR_BIKE_DATA_UUID(0x2AD2),
    FITNESS_MACHINE_FEATURE_UUID(0x2ACC), FITNESS_MACHINE_STATUS_UUID(0x2ADA),
    FITNESS_MACHINE_CONTROL_POINT_UUID(0x2AD9)
  IBD_FLAG: Indoor Bike Dataのビットフラグ（MORE_DATA以下13ビット、
    ビット0は「立っていれば瞬間速度が“ない”」という逆論理点に注意）
  CPS: CPS_SERVICE_UUID(0x1818), CYCLING_POWER_MEASUREMENT_UUID(0x2A63),

6-5. cadence.js
────────────────────────────────────────
  createCadenceCalculator({staleTimeoutMs=2000}={}) →
    (revolutions, eventTime) => cadenceRpm

  CPSのクランク累積回転数・イベント時刻（1/1024秒単位、64秒で
  ラップアラウンド）から差分を取り、
    cadenceRpm = deltaRevolutions × 60 × 1024 / deltaEventTime
  で算出するクロージャ。差分イベントが無い場合は直近値を保持しつつ、
  staleTimeoutMs（既定2000ms）以上更新が無ければ0を返す
  （停止検出）。uint16ラップアラウンドは `(cur-prev+0x10000)&0xFFFF` で
  安全に処理する。

7-2. route.js

7-3. physics.js
────────────────────────────────────────
  G_MS2 = 9.81 (m/s²)　　RHO_KGM3 = 1.225 (kg/m³)
  DEFAULT_PHYSICS = { massKg: 80, cdA: 0.32, crr: 0.005 }

  altitudeFactor(elevM) → number
    max(0, 1 - max(0, elevM-1500) * 0.0001)
    （1500m以下は1.0、以降100mごとに約1%低下）

  stepVelocity(powerW, gradientPercent, currentVelocityMs, dtSec,
    {massKg, cdA, crr}) → number（新しい速度 m/s）
    theta = atan(gradientPercent/100)
    vEff  = max(currentVelocityMs, 0.5)  // 発進時のゼロ割回避
    pRolling  = crr·massKg·G_MS2·cos(theta)·vEff
    pAir      = 0.5·RHO_KGM3·cdA·vEff³
    pGravity  = massKg·G_MS2·sin(theta)·vEff
    accel = (powerW - pRolling - pAir - pGravity) / (massKg·vEff)
    return max(0, currentVelocityMs + accel·dtSec)

7-4. torque.js
────────────────────────────────────────
  calcTorque(powerW, cadenceRpm) → number|null
    T[N·m] = P[W] × 60 / (2π × cadence[rpm])
    cadenceRpm<20 または powerW<=0 のとき null
    （近似が不安定になる領域のガード）

7-5. simulator.js
────────────────────────────────────────
  class RideSimulator（route.jsとphysics.jsの橋渡し。タイマー・BLE・
  UIには関与しない純粋シミュレーション。tickループの主導権は
  ride/rideController.jsが持つ）

    constructor(route: Route, params)
    set altitudeEffectEnabled(bool)
    tick(powerW, dtSec)
      gradient = route.getGradientAt(s.distanceM)
      elevM    = route.getElevationAt(s.distanceM) ?? 0
      factor   = altitudeEffectEnabled ? altitudeFactor(elevM) : 1
      effectivePowerW = powerW × factor   ← 高度によるパワー低下は
                                             ここで速度計算に反映される
      newV = stepVelocity(effectivePowerW, gradient, s.velocityMs,
                           dtSec, params)
      s.distanceM = min(prevDist + newV×dtSec, route.totalDistanceM)
      elevationGainM は正の標高差のみ加算
    getState() → {distanceM, velocityMs, elapsedSec, elevationGainM,
      altitudeFactor, currentLat, currentLon, currentGradientPercent,
      headingDeg}
      headingDeg: 最大10m先読みしatan2(Δlon, Δlat)×180/π（真北基準
      時計回り）で算出、残距離僅少時は0
    get isFinished() → distanceM >= route.totalDistanceM
    getSimState() / restoreSimState({...}) → チェックポイント保存・復元
      （restoreSimStateは自動的にpause状態にする。再開は呼び出し元が
      明示的にresume()する）
    pause() / resume() / reset()

11. 実写風景写真（src/mapillary/）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  api.js:
    fetchCandidateImages(point) — {lat,lon}中心に半幅35m
      （bboxFromPoint）のbboxでGraph API `/images` をGET
      （fields: id, compass_angle, computed_compass_angle, is_pano,
      sequence_id, quality_score, captured_at, thumb_1024_url, creator。
      limit=40）。トークンはVITE_MAPILLARY_TOKEN。
    fetchThumbUrl(imageId) — サムネイルURLは長期キャッシュ不可のため、
      表示直前に単体エンティティを再取得して新鮮なURLを得る。

  bearing.js:
    precomputeBearings(points, window=2) — 各点に前後window点を使った
      大圏方位角(bearing)を付与。逆走時は反転済みpoint配列に対して
      呼ぶこと。

  score.js:
    effectiveAngle(image) — computed_compass_angle優先、なければ
      compass_angle。
    selectBest(images, routeBearing, activeSequenceId=null) —
      角度差>45°は除外(score=-1)。パノラマは
      `35 - diff×0.2 + quality×20 + recency`、通常写真は
      `50 - diff×0.3 + quality×20 + recency` を基礎スコアとし、
      activeSequenceId一致で+15（SEQUENCE_BONUS、直前と同じ撮影
      シーケンスの継続を優先）。最高スコアの1件を返す（無ければnull）。

  cache.js:
    IndexedDB `mapillaryCache` ストア(KV)。CACHE_VERSION=7,
    STALE_MS=90日。
    resolveImageForPoint(cachePrefix, idx, point, activeSequenceId) —
      キー `${cachePrefix}::${idx}`。有効キャッシュがあれば
      サムネイルURLのみ再取得して返す（stripUrl）。無ければ
      fetchCandidateImages+selectBestを実行しキャッシュへ保存。

  lookahead.js:
    class ActiveIndexTracker(points) — update(currentDistanceM)で、
      次点のdistanceFromStartM-25mを現在距離が超えたら単調に
      activeIndexを進める（25m手前倒しトリガー）。
    class MapillaryLookahead(cachePrefix, points, lookaheadPoints=6) —
      Map<index,{status,image,routeBearing}>のバッファを持ち、
      tick(activeIndex)でactiveIndex+lookaheadPoints
      （既定6点≈想定50m間隔で約300m先）まで1件ずつ先読み取得
      （#inFlightで同時1件に制限）。activeIndex-1より古いバッファは
      破棄。#activeSequenceIdを跨ぎ保持し継続性ボーナスへ利用。
      getStateFor(index) → 未取得時は{status:'pending', image:null,
      routeBearing:null}。resolveImageForPointの解決直後に
      panel.jsのpreloadImage(thumb_1024_url)を呼び、表示（activeIndex
      到達）より前にブラウザへ画像バイトを先読みさせる（切り替え時の
      黒画防止）。

  panel.js:
    preloadImage(url) — new Image()にurlを割り当てるだけの先読み
      ヘルパー。lookahead.jsのtick()から、画像が先読みバッファに
      入った時点（表示のおよそ300m/lookaheadPoints分前）で呼ばれる。
    updatePhotoPanel(status, image, routeBearing=null, distanceM=null)
      — status==='pending'は無視（チラつき防止）。画像なしは
      "No Image"。表示直前の再チェックとしてDISPLAY_MAX_DIFF_DEG=25
      （キャッシュ選定時の45°より厳しい）を超えたら"No Image"に
      フォールバック。パノラマはtransform-origin＋2倍ズームで
      routeBearing方向を画面中央にクロップ。撮影者クレジット
      「© {username} / Mapillary」を表示。
    resetPhotoPanel() — プレースホルダーに戻す。

  end-to-end フロー（main.jsのonRouteSelectedコールバック～
  rideController.js #tick、両者にまたがる）:
    ルート選択時: MAPILLARY_ENABLED（VITE_MAPILLARY_TOKEN有無）が
      trueなら precomputeBearings(route.points) を1回実行、
      cachePrefix = `${routeId}:${reversed?'r':'f'}` で
      MapillaryLookahead/ActiveIndexTrackerを構築、初回
      lookahead.tick(0) で1枚目をプレビュー。
    ライド中: rideController.js #tick 内で
      tracker.update(distanceM)→activeIdx、
      lookahead.tick(activeIdx)（非同期）、
      lookahead.getStateFor(activeIdx) を updatePhotoPanel に渡す。

12. FIT生成（src/export/fitWriter.js）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

外部FITライブラリを使わず、バイナリを直接組み立てる自前実装。

  FIT_EPOCH_MS = Date.UTC(1989,11,31)
  グローバルメッセージ番号: FILE_ID=0, ACTIVITY=34, SESSION=18,
    LAP=19, RECORD=20
  ベース型: ENUM=0x00, UINT8=0x02, UINT16=0x84, SINT32=0x85,
    UINT32=0x86

  class ByteWriter（非export）— u8/u16/u32/s32のリトルエンディアン
    書き込みヘルパー、.bytesでUint8Array取得
  fitCrc(bytes) — FIT標準のCRC-16（nibbleテーブル方式）
  writeDef(w, localType, globalNum, fields) — FIT定義メッセージ書き込み
  fitTs(ms) → FITタイムスタンプ（FIT epoch起点の秒）
  semicircles(deg) → round(deg × 2^31/180)（緯度経度のFIT表現）
  altRaw(m) → round((m+500)×5)、[0,0xFFFE]でクランプ、null時0xFFFF(invalid)
  avg(arr, fn) → fn(x)>0の要素のみで平均

  buildFit(summary) → Uint8Array（GPS付きルートライド用）
    summary: {routeName, startedAt, endedAt, samples:[{timestampMs,
      lat, lon, elevationM, distanceM, velocityMs, powerW, cadenceRpm,
      heartRateBpm}]}
    メッセージ順: FileID → Record(サンプル毎、緯度経度はsemicircles、
      標高はaltRaw、距離は×100、速度は×1000) → Lap
      （event=9/event_type=1固定値、total_elapsed_timeはミリ秒を
      そのままuint32格納 = FIT仕様上のスケール1000秒フィールドと
      解釈上整合） → Session（sport=2/サイクリング,
      sub_sport=58/バーチャル活動、平均値、total_ascent=calcAscent()）
      → Activity（event=26/event_type=1）
    ヘッダー14バイト + データ + データ部CRC(2バイトLE) で構成
    （buildHeader()）。routeNameは文字列としては書き込まれない
    （呼び出し元がStravaアップロード時の名前として別途使用）。

  buildWorkoutFit(summary) → Uint8Array（GPS無し、ワークアウト用。
    Stravaへtrainer=1でアップロードする想定）
    summary: {workoutName, startedAt, endedAt, samples:[{timestampMs,
      powerW, cadenceRpm, heartRateBpm, distanceM?, velocityMs?}]}
    Recordから緯度経度・標高を除外、Session.sub_sport=6
    （インドアサイクリング）、total_ascentフィールドなし。それ以外は
    buildFit()と同じメッセージ構成。

  buildHeader(dataSize) — サイズ14, protocol=0x20, profile version=2132,
    データサイズ(u32 LE), ".FIT"マジック, 先頭12バイトのCRC(u16 LE)
  calcAscent(samples) — 連続サンプル間の正の標高差の合計

13-1. src/strava/auth.js（クライアント側）
────────────────────────────────────────
  startAuthorization() — `https://www.strava.com/oauth/authorize` へ
    window.location.href でリダイレクト。パラメータ:
    client_id, redirect_uri, response_type=code,
    scope='activity:write,read', approval_prompt=auto。
    PKCE不使用（クライアントシークレットはサーバー側関数に隠蔽する
    標準の認可コードフロー）。

  exchangeCode(code) → tokens
    同一オリジンの `/strava-exchange`（Cloudflare Workers、
    src/worker/index.js）へ{code}をPOST。レスポンスから
    {accessToken, refreshToken, expiresAt(unix秒), athleteId,
    athleteName} を組み立て、settings.stravaTokensへ保存。

  getValidAccessToken() → accessToken
    expiresAt > now+60秒 ならキャッシュ済みトークンをそのまま返す。
    それ以外は `/strava-refresh` へ{refresh_token}をPOSTし、
    access_token/expires_atのみ更新して保存
    （※ Stravaがrefresh_tokenをローテーションした場合、レスポンスの
    新refresh_tokenを保存していない実装になっている点に注意
    ―将来的な改善候補）。

  getConnectionInfo() → {athleteName, athleteId}|null（リフレッシュなし）
  disconnect() → settings.stravaTokens削除

13-2. src/worker/index.js（サーバー側、単一Workerスクリプト）

13-3. src/strava/upload.js

────────────────────────────────────────

14. PWA / Service Worker
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  public/manifest.json:
    name: "Virtual Cycling", short_name: "VCycl",
    display: "standalone", orientation: "landscape",
    background_color: "#1a1a2e", theme_color: "#0f3460",
    icons: 192px/512px（purpose: any maskable）

  public/sw.js:
    CACHE_NAME = 'vcycl-v{N}'（sw.js自体の内容を変更するたびにNを
      インクリメントすること。インクリメントを忘れると、ブラウザは
      sw.js自体のバイト差分を検知できずservice workerを再インストール
      しないため、ハッシュ付きJSファイル名が変わるようなアプリ側の
      更新が永久に反映されない事故につながる。実装時に実際に
      この事故が発生し、デプロイ後もアプリが古いバージョンのまま
      （存在しないハッシュのJSを参照して起動失敗）になったことがある）
    PRECACHE_URLS: '/', manifest.json, favicon類, icons/*
    install: PRECACHE_URLSをキャッシュ、skipWaiting()
    activate: 旧キャッシュ削除、clients.claim()
    fetch:
      - ナビゲーションリクエスト（event.request.mode === 'navigate'、
        HTML本体）: ネットワーク優先。失敗時（オフライン）のみ
        キャッシュ、それも無ければ '/' のキャッシュにフォールバック
      - それ以外（ハッシュ付きJS/CSS・アイコン等）: cache-first
        （ファイル名にコンテンツハッシュを含むため、キャッシュ優先で
        問題ない）

  登録: index.html末尾のインラインscriptで
    `navigator.serviceWorker.register('/sw.js')`（main.jsではなく
    index.html側に直接記述されている点に注意）。

  __APP_VERSION__: vite.config.local.js / vite.config.js で
    package.jsonのversionを読み込みビルド時定数として注入。
    main.jsが `#app-version` 要素に 'v' + __APP_VERSION__ を表示。
    オーナーモード時は隣接する `#owner-badge` に "(o)" が表示される
    （詳細は13-4節）。

15. Cloudflare Workers デプロイ設定
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  現状（2026-07-11時点）: GitHub連携（Workers Builds）設定済み。
  `main`ブランチへのpushで自動ビルド・デプロイされる
  （ビルドコマンド: npm run build、デプロイコマンド: npx wrangler deploy）。
  Cloudflareダッシュボードの「Settings → Build」にビルド時変数
  （VITE_*、7変数）、「Settings → Variables & Secrets」に実行時
  シークレット（4変数）を設定済み。

  手動デプロイ（15-2〜15-4）は、ローカルでの動作検証や、Workers Builds
  側の自動デプロイが失敗した際の復旧手段として残す。通常の変更反映は
  git push のみで完結する。

  ⚠ 15-3[1]で判明した「`wrangler secret put`だけではシークレットが
  反映されないことがある」問題が、Workers Builds のデフォルトデプロイ
  コマンド（`npx wrangler deploy`、--secrets-fileなし）でも起こり得るか
  は未検証。自動デプロイ後は `GET /api/auth?owner=xxx` 等で実行時
  シークレットが正しく効いているか確認すること。もし同じ問題が
  再発する場合は、Cloudflareダッシュボードの「Variables & Secrets」
  パネルでの設定（プラットフォームネイティブの経路）が正しく機能して
  いるかを疑うこと。

15-1. wrangler.jsonc
────────────────────────────────────────
  {
    "name": "gpx-vcycl",
    "compatibility_date": "2026-07-11",
    "main": "./src/worker/index.js",

15-2. 初回セットアップ手順
────────────────────────────────────────
  1. npx wrangler login              （ブラウザでCloudflareアカウント認証）
  2. .dev.vars を作成し、4章のサーバー側専用4変数を記入

────────────────────────────────────────
15-3. 既知の落とし穴（実装時に実際に発生した問題）
────────────────────────────────────────
  [1] `wrangler secret put <NAME>` を個別に実行しただけでは、後続の
      `npm run deploy`（vite build && wrangler deploy）でシークレットが
      Worker実行時に反映されないことがあった（@cloudflare/vite-plugin が
      生成する「リダイレクトされたwrangler設定」経由のデプロイ特有の
      挙動と推測される。原因は完全には特定できていない）。
      → 対策: `wrangler deploy --secrets-file .dev.vars` で、コード
        デプロイと同じタイミングでシークレットを明示的に添付する。
        このフラグは「ファイルに含まれないシークレットは前バージョンから
        保持される」ため、繰り返し実行しても安全（追記のみ・非破壊的）。

  [2] `.dev.vars` で新規Workerに初めてシークレットを設定しようとすると、
      「Worker "gpx-vcycl" が存在しません。作成しますか？」という
      確認が出る（非対話実行時は自動でyes扱いになる）。これは想定通り
      の動作で、空のWorkerが作成された後、実コードのデプロイで
      上書きされる。

  [3] VITE_STRAVA_REDIRECT_URI のようなビルド時変数は、ローカル開発
      （LAN IP）と本番（Workersの公開URL）で値が異なる。.env に開発用の
      値を書いたままにすると、本番ビルドにも開発用URLが焼き込まれ、
      Strava認可画面で `{"errors":[{"field":"client_id","code":"invalid"}]}`
      のようなエラーになる（client_idそのものは正しくても、
      redirect_uriやビルドタイミングの環境変数未定義が原因で
      client_idが `undefined` 文字列として送信されてしまうケースが
      あった）。
      → 対策: `.env.production` を作成し、本番URLを使う変数
        （VITE_STRAVA_REDIRECT_URI 等）だけをそこに上書き定義する。
        `vite build` はデフォルトで production モードのため
        `.env` → `.env.production` → `.env.local` の順で読み込み・
        マージされる。デプロイ前に、生成されたバンドル内に
        期待する値が実際に含まれているか確認すること
        （例: `curl <デプロイURL>/assets/index-*.js | grep <期待する値>`）。

────────────────────────────────────────
15-4. 通常の反映方法（Workers Builds経由）
────────────────────────────────────────
  main ブランチへ git push するだけでよい。Workers Builds が
  自動的に npm run build → npx wrangler deploy を実行する。
  デプロイ完了後は動作確認すること（例: GET /api/auth?owner=xxx が
  200を返すか）。

  ローカルから手動で反映したい場合（緊急復旧・検証用途）:
    npm run deploy
      package.json: "deploy": "vite build && wrangler deploy"
      ※ シークレットに変更がない場合はこれで十分だが、15-3[1]の問題が
        再発する可能性があるため、デプロイ後は必ず動作確認すること。
        シークレットを変更した場合は15-2の手順4
        （--secrets-file 付き）を使うこと。

19. 招待制ログイン（Supabase Auth・メールOTP）の実装
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  gpx-navi・gpx-editor（同じSupabaseプロジェクトを共有する姉妹アプリ）と
  同じパターンで実装する。spec.txt 9-5節の機能仕様を参照。

  【変更履歴・2026-07-18】当初はマジックリンク方式（メール内リンクを
  クリックしてログイン）で実装したが、以下の問題が判明したため
  8桁コード手入力方式（verifyOtp）に変更した:
    - PKCE flow（supabase-js既定）はリンクをリクエストしたブラウザの
      localStorageにcode verifierを保存し照合するため、別ブラウザ/
      デバイスでリンクを開くと認証に失敗する
    - iPhoneのGmailアプリ等、メールクライアントのリンクプレビュー・
      長押しコピー機能がリンクを自動プリフェッチしてしまい、
      ワンタイムリンクが本人のクリック前に無効化されてしまう
    - コード手入力方式であれば、人間が手で入力する必要があるため
      自動プリフェッチで消費されず、どのブラウザ・デバイスでも使える
    - 同じSupabaseプロジェクトを共有するgpx-editor・gpx-naviと同時に
      対応済み。Email Templates（Magic Link）から{{ .ConfirmationURL }}
      を削除し{{ .Token }}のみを案内する内容に変更済みのため、
      本アプリ側の追加ダッシュボード作業は不要

────────────────────────────────────────
19-1. src/supabase/client.js
────────────────────────────────────────
  isSupabaseConfigured() → boolean
    VITE_SUPABASE_URL と VITE_SUPABASE_ANON_KEY の両方が設定されて
    いるかを返す。未設定時はヘッダーの歯車アイコン自体を非表示にする
    （main.js側）。

  getSupabaseClient() → SupabaseClient
    シングルトンのSupabaseクライアント（@supabase/supabase-js の
    createClient）。storage/remoteRoutes.js もこの関数を経由して
    同一クライアントを使う（従来remoteRoutes.js内で独自に
    createClientしていたのをここに集約、クライアントの二重生成を
    避ける）。

  ※ VITE_SUPABASE_ANON_KEY には必ずpublishable key（sb_publishable_...）
    を使うこと。secret key（sb_secret_...）を誤って設定すると
    クライアントバンドルに焼き込まれ、RLSを無視した全権限が
    世界に公開されてしまう（実装時に実際にこの取り違えが発生した
    ため特記する）。

────────────────────────────────────────
19-2. src/supabase/auth.js
────────────────────────────────────────
  sendMagicLink(email) → {ok: true} | {ok: false, error}
    supabase.auth.signInWithOtp({email}) を呼ぶ。サインアップ
    （signUp）は呼ばない＝招待制（Supabase側で事前登録された
    メールアドレスのみログインできる）。emailRedirectToは指定しない
    （リンククリックを使わないコード入力方式のため不要）。

  verifyOtp(email, token) → {ok: true} | {ok: false, error}
    supabase.auth.verifyOtp({email, token, type: 'email'}) を呼ぶ。
    メールで届いた8桁コードを検証し、成功するとセッションが確立される。

  getSession() → Session|null
    supabase.auth.getSession() のラッパー。

  onAuthStateChange(callback: (session) => void) → unsubscribe関数
    supabase.auth.onAuthStateChange() のラッパー。verifyOtp成功時の
    セッション確立をこの購読経由で検知する。

  signOut() → void

19-3. main.js での配線
────────────────────────────────────────
  DOM要素:
    #auth-gear-btn      ヘッダー右上の歯車アイコンボタン
    #auth-popover        ポップオーバー本体（main.jsがinnerHTMLを都度再構築）
    #db-status-error     DB初期化失敗時のみ表示するエラーテキスト
                          （旧#db-statusを置き換え。「DB初期化済み」の
                          成功メッセージ表示は廃止）

  initAuth()（init()内、initDb()成功後に呼ぶ）:
    1. isSupabaseConfigured() が false なら #auth-gear-btn を隠して終了
    2. getSession() で初期状態を取得 → currentUser にセット →
       updateAuthUI()
    3. onAuthStateChange() を購読 → currentUser 更新のたびに
       updateAuthUI() + renderAuthPopover()
    4. #auth-gear-btn クリックで #auth-popover の open クラスを
       トグル（開くたびに renderAuthPopover() で中身を再構築）
    5. ポップオーバー外クリックで自動的に閉じる
       （document への click リスナーで判定）

  updateAuthUI():
    ログイン状態に応じて #auth-gear-btn に .logged-in クラスを付与、
    #load-remote-gpx-btn と #load-remote-zwo-btn の disabled を
    切り替える（ログイン時のみ有効）。ゲーティング対象はこの2ボタン
    のみ（spec.txt 9-5節）。

  renderAuthPopover()（3状態、authPendingEmailで送信済みメールを保持）:
    - ログイン中: 「ログイン中: {email}」+ ログアウトボタン + キャンセルボタン
    - コード送信済み（authPendingEmail有り）: 案内テキスト + 8桁コード
      入力欄 + 「ログイン」ボタン（verifyOtp呼び出し）+ 「戻る」ボタン
      （authPendingEmailをnullに戻して再描画）+ ステータステキスト
    - 未ログイン・未送信: メール入力 + 「ログインコードを送る」ボタン
      （sendMagicLink呼び出し、成功したらauthPendingEmailにセットして
      再描画）+ キャンセルボタン + ステータステキスト
    キャンセルボタンは各状態共通のmakeAuthCancelBtn()ヘルパーで生成し、
    クリックで#auth-popoverのopenクラスを外すだけ（状態変更なし）。

19-4. 外部設定（Supabaseダッシュボード側、アプリコード外）
────────────────────────────────────────

  - Authentication → Email Templates → Magic Link のテンプレートから
    {{ .ConfirmationURL }}（クリック用リンク）を削除し、{{ .Token }}

  - 招待ユーザーのメールアドレスは Authentication → Users で

    ログインコードを送っても実際にはログインできない）
