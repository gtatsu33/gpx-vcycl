1. アプリケーション概要
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

iPad Safari では Web Bluetooth が使えないため「Bluefy」ブラウザ上で動作する
バーチャル室内サイクリング PWA。GPXルートを読み込み、スマートトレーナー
（Saris等、FTMS対応機）で勾配を再現しながら、実在のコースを仮想的に走る。

主な機能:
  - スマートトレーナー / パワーメーター / 心拍計の BLE 接続・管理
  - GPXルートの読み込み（ローカルファイル／リモートSupabaseストレージ）・保存・逆走
  - ルートに沿った仮想走行（物理シミュレーション、3Dマップ表示、2Dマップ表示、
    実写風景写真表示、HUD、コース全体標高表示）
  - トレーナーへの勾配フィードバック（負荷制御）
  - ZWO形式ワークアウトの読み込みとERGモード実行（ルート非依存）
  - ライド記録の保存、FIT形式へのエクスポート、Stravaへのアップロード
  - ライド履歴の閲覧・再アップロード・削除

利用者: 開発者本人のみ（シングルユーザー前提）。認証は Strava 連携時のみ発生する。

対象デバイス: iPad（iPadOS 16+）+ Bluefy ブラウザ。開発・デバッグは
デスクトップ Chrome/Edge でも可能（Web Bluetooth 部分はモックで代替）。

2. 画面構成の全体像
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

アプリはタブ切り替え式の単一ページ構成。

  ┌─────────────────────────────────────────────┐
  │  ヘッダー（アプリ名・バージョン・ログイン用歯車アイコン）│
  ├─────────────────────────────────────────────┤
  │  タブ: [デバイス] [ルート] [ワークアウト] [履歴] │
  ├─────────────────────────────────────────────┤
  │  タブ内容（後述 3〜6章）                        │
  └─────────────────────────────────────────────┘

  加えて、以下はどのタブからもオーバーレイ表示される:
    - 設定モーダル（7章）
    - Strava未接続確認ダイアログ（ライド開始前）
    - ライド終了モーダル／ワークアウト終了モーダル（6章）
    - ルート名入力モーダル、リモートGPX選択モーダル

  画面向き: ランドスケープ（横向き）を前提とする。

3. デバイスタブ（BLE接続管理）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

────────────────────────────────────────
3-1. 概要
────────────────────────────────────────

3種類の BLE デバイスを個別に接続・管理する。1台ずつのみ接続可能
（同一種類のデバイスを複数同時接続する機能はない）。

  - トレーナー（FTMS: Fitness Machine Service）— 速度・ケイデンス・パワーの
    取得、および勾配に応じた負荷制御（書き込み）に対応
  - パワーメーター（CPS: Cycling Power Service）— パワー・ケイデンスの取得のみ
  - 心拍計（HRS: Heart Rate Service）— 心拍数の取得のみ

────────────────────────────────────────
3-2. 画面構成
────────────────────────────────────────

  - 上部: 現在有効なパワー値の大表示（ソース名を併記）
  - デバイスカード×3（トレーナー／パワーメーター／心拍計）:
    - 接続状態バッジ（未接続／接続中.../接続済）
    - 現在値表示（速度・ケイデンス・パワー、または心拍）
    - 接続／切断ボタン
  - トレーナーカードのみ「ダミー」チェックボックスを持つ（後述 3-4）
  - BLE通信ログ（タイムスタンプ付き、直近ログを上に追加）
  - Settingsボタン（設定モーダルを開く）

────────────────────────────────────────
3-3. パワーソース選択
────────────────────────────────────────

トレーナーとパワーメーターを両方接続した場合、どちらの値を
シミュレーションのパワー入力として使うかを設定モーダルで選択する
（「トレーナー」または「パワーメーター」）。

  - 選択切替時、直前の値はリセットされる（0にクリア）
  - 心拍はソースに関わらず心拍計から取得した値を常に使用する
  - ケイデンスは選択中のパワーソースに紐づく値を使用する

────────────────────────────────────────
3-4. ダミートレーナーモード
────────────────────────────────────────

実機トレーナーが無い環境（開発時・デモ時）向けに、固定値
（設定FTP値のパワー、ケイデンス90rpm固定、心拍は実測があれば使用）を
生成するモード。ライド開始前の確認で「Strava未接続警告」をスキップする
（記録・アップロードを試みない前提の動作確認用モード）。

────────────────────────────────────────
3-5. モックモード（開発補助）
────────────────────────────────────────

URLに `?mock=1` を付与すると、3種のBLEクライアントがすべてモック実装に
差し替わり、実機なしで疑似的な変動データ（サイン波＋ノイズ）が自動生成
される。UI・シミュレーションロジックの動作確認に用いる。

────────────────────────────────────────
3-6. 異常系
────────────────────────────────────────

  - デバイスが見つからない／接続拒否 → ログに接続エラーを表示、ボタンは
    「接続」に戻る
  - ライド中にBLE切断 → ログに切断を表示。直前の値を保持したまま表示は
    更新が止まる
  - パワーソース切替直後 → パワー・ケイデンス表示は一時的に0にリセット

4. ルートタブ
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

────────────────────────────────────────
4-1. GPXルートの読み込みと管理（事前ライド画面・共通）
────────────────────────────────────────

【読み込み手段】
  - ローカルファイル: 端末上のGPXファイルを選択して読み込む（認証不要）
  - リモート: あらかじめ登録済みのGPXファイル一覧（表示名・距離・獲得標高
    付き）から選択してダウンロードする（外部ストレージ経由。DB仕様は
    db.v4.md参照）。**招待ユーザーとしてログイン済みの場合のみ利用可能**
    （9-5節参照）。未ログイン時はボタンが無効化され、押しても反応しない

  いずれの手段でも、読み込み後にルート名の確認・編集ダイアログを表示し、
  保存してからルート一覧に追加する。

【保存済みルート一覧】
  各ルートについて以下を表示・操作できる:
    - ルート名、総距離、累積標高
    - 「逆走」チェックボックス（選択中ルートに対し、始点⇔終点を反転して
      走行する。名前付きウェイポイントも逆順で有効なまま扱われる）
    - 削除ボタン（確認ダイアログあり）

  ルートを選択すると、地図上にルート全体が描画され、標高プロファイル
  （小型グラフ）がプレビュー表示される。

────────────────────────────────────────
4-2. 事前ライド画面（ライド開始前）
────────────────────────────────────────

画面左側にルート一覧・標高プロファイル・「ライドを開始」ボタンを配置する
サイドパネルを表示し、画面右側にプレビュー用の地図エリアを表示する。

  地図エリアの構成（この画面のみ）:
    - 上段: 3Dマップ（走行風景のプレビュー、選択中ルートの始点付近を表示）
    - 下段: 実写風景写真（該当ルートの1枚目）と2Dマップ（ルート全体）を
      横並びで表示

  「ライドを開始」ボタンはルート未選択時は無効。押下すると、Strava未接続
  であればライド開始前に確認ダイアログを表示する（ダミートレーナー時は
  スキップ）。

  本画面のレイアウトは今後の変更対象外とし、変更する場合は別途仕様化する。

────────────────────────────────────────
4-3. ライド中画面
────────────────────────────────────────

ライド開始と同時に、事前ライド用のサイドパネルは非表示になり、
地図エリアが画面全体まで拡大する。画面全体を縦2段に分割する。

  ┌─────────────────────────────────────────────┐
  │  上段（可変高さ、横方向を左右2エリアに分割）      │
  │  ┌───────────────────────┬─────────────────┐ │
  │  │                       │  実写風景写真（上半分）│
  │  │   3Dマップ（走行風景） │───────────────── │ │
  │  │   + HUDオーバーレイ     │  2Dマップ（下半分） │ │
  │  │   （左 2/3）            │  （右 1/3）         │ │
  │  └───────────────────────┴─────────────────┘ │
  ├─────────────────────────────────────────────┤
  │  コース全体標高表示（フル幅・常時表示・固定）    │
  ├─────────────────────────────────────────────┤
  │  ライドコントロール（一時停止／終了／トレーナーリセット）│
  └─────────────────────────────────────────────┘

  左右比率は概ね 2:1。コース全体標高表示はオーバーレイではなく独立した
  帯として常時表示し、現在位置をマーカーで示す。

【3Dマップ（走行風景）】
  進行方向を一人称視点で描画する。路面の色は勾配に応じて変化する
  （下記グラデーション区分を共通で使用）:
    < 3%   : 緑（平坦〜緩い上り／下り）
    3〜6%  : 黄（中程度の上り）
    6〜9%  : 橙（きつい上り）
    9〜12% : 赤（非常にきつい上り）
    ≥12%  : 濃赤茶（激坂）
  1kmごと、および名前付きウェイポイント（GPX中で「」括弧付きの名前を
  持つ地点）ごとに、残り距離を示す看板を仮想的に設置する。ゴール地点には
  専用の看板を設置する。

【HUDオーバーレイ（3Dマップ上に半透明パネルで重畳表示）】
  情報を3グループに分けて配置する。文字ラベルは中央上部のみ省略し、
  アイコンと単位表記で区別する。

  中央上部＝リアルタイム系（2段構成）:
    上段（大きめ）: パワー[W]、ケイデンス[rpm]、心拍[bpm]
    下段（小さめ）: 速度[km/h]、トルク[N·m]
    ※ パワーが標高効果（4-5節）で低下している場合、低下率を併記する

  左＝累積系:
    経過時間、走行距離（残り距離を併記）、累積標高[m↑]

  右＝ルート情報:
    現在の勾配[%]（上記色分けルールに準じたバッジ表示）、標高[m]

  左カラム下部＝直近1kmの勾配プロファイル（オーバーレイ）:
    現在地点から1km先までの標高・勾配を色分けグラフで表示。500m・1km
    地点にラベルを表示し、現在地点を左端の縦線マーカーで示す。

  ゴール到達時は「ゴール！」のメッセージを表示する。

【実写風景写真（右上）】
  現在地点付近で撮影された実写のストリートレベル写真を、走行の進行に
  合わせて自動的に切り替えながら表示する（4-7節）。写真が見つからない
  場合は「No Image」のプレースホルダーを表示する。撮影者クレジットを
  写真下部に表示する。

【2Dマップ（右下）】
  ルート全体（青線）と走行済み区間（緑線）、現在位置（進行方向を示す
  矢印マーカー）を表示する。地図をユーザーがスワイプ等で操作すると
  自動追従が解除され、再センターボタンを押すと現在地に戻り自動追従が
  再開する。

【コース全体標高表示（最下部）】
  ルート全体の標高プロファイルを距離軸で表示し、現在位置を縦線マーカーで
  示す。距離ラベル（10分割）を表示する。

【ライドコントロール】
  - 一時停止／再開ボタン
  - 終了ボタン（走行を終了し、ライド終了モーダルを表示）
  - トレーナー緊急リセットボタン（トレーナー負荷制御が想定外の高負荷に
    なった場合の安全弁。FTMSリセットコマンドを送信する）

────────────────────────────────────────
4-4. 物理シミュレーション
────────────────────────────────────────

パワー入力（BLEから取得、直近数秒の移動平均で平滑化）と、ルートの
現在地点の勾配から、リアルタイムに走行速度をシミュレーションする。

  力の釣り合いモデル（1ステップ = 短い時間刻み dt 秒ごとに速度を更新）:
    転がり抵抗パワー = Crr × 質量 × 重力加速度 × cos(勾配角) × 速度
    空気抵抗パワー   = 0.5 × 空気密度 × CdA × 速度³
    重力パワー       = 質量 × 重力加速度 × sin(勾配角) × 速度
      （勾配角は「勾配% ÷ 100」のアークタンジェント）
    加速度 = (入力パワー − 転がり抵抗パワー − 空気抵抗パワー − 重力パワー)
             ÷ (質量 × 速度)
    新しい速度 = max(0, 現在速度 + 加速度 × dt)

  パラメータ（設定モーダルで調整可能。既定値）:
    質量        : ライダー体重 + 自転車重量（既定 合計80kg）
    CdA         : 前面投影面積×抗力係数（既定 0.32 m²）
    Crr         : 転がり抵抗係数（既定 0.005）
    重力加速度   : 9.81 m/s²（固定）
    空気密度     : 1.225 kg/m³（標準大気、固定）

  速度が停止に近い領域では、実効速度に下限（約1.8km/h相当）を設けて
  発進時の挙動を安定させる。

  走行距離・累積標高・経過時間はシミュレーション結果から積算する。

────────────────────────────────────────
4-5. 高度によるパワー低下（オプション、既定ON）
────────────────────────────────────────

標高1500mを超える区間では、高度による有酸素能力低下を模した
パワー低下係数を入力パワーに乗じてからシミュレーションする。
標高100mごとに約1%低下する（例: 標高3000mでは約15%低下）。
1500m以下では低下しない。設定でON/OFFを切り替え可能。

────────────────────────────────────────
4-6. トレーナー負荷制御
────────────────────────────────────────

対応トレーナー（FTMS Control Point対応機）に対し、現在地点の勾配を
定期的に送信し、実際のペダリング抵抗を変化させる。

  - 設定でON/OFF切替可能（既定ON、要FTMS接続）
  - 実際にトレーナーへ送る勾配 = GPX上の勾配 × 負荷強度（設定、既定0.5）
    ※フルの勾配だとトレーナーへの負荷が過大になるための緩和係数
  - 勾配は ±25% にクリップして送信（トレーナー保護）
  - 送信頻度: 1秒ごと、または勾配が前回送信値から±1%以上変化した時点で
    即時送信
  - トレーナーがControl Point非対応の場合は、負荷制御なしで速度・パワー
    読み取りのみ継続する（読み取り専用フォールバック）
  - ライド終了時・緊急リセット時は負荷をゼロに戻す

────────────────────────────────────────
4-7. 実写風景写真（ルックアヘッド表示）
────────────────────────────────────────

外部の街並み写真データベースから、走行位置に応じた実写画像を
自動選択して表示する機能（要 外部トークン設定、未設定時は機能自体を
無効化する）。オーナーモードでGoogleMaps/Street Viewを選択している間は、
この機能自体を無効化する（9-4節参照。Street Viewと排他）。

  - ルート選択時に、ルート全体の進行方位（bearing）を事前計算する
  - 走行中、現在地点から少し先（数百m先まで）を先読みして候補画像を取得し、
    バッファリングしておく（表示の滑らかさのため）
  - 候補画像は、ルート進行方位との角度差が一定以内（±45°程度）のものに
    限定し、角度の近さ・画質スコア・撮影の新しさ・直前と同じ撮影シーケンス
    かどうか（継続性ボーナス）で最良の1枚を選ぶ
  - 表示直前にも、より厳しい角度しきい値（±25°程度）で再チェックし、
    ずれが大きい場合は「No Image」にフォールバックする
  - パノラマ画像の場合は、進行方位を画面中央に来るようにクロップ・拡大
    して表示する
  - 一度取得した画像情報はローカルにキャッシュし、一定期間（数十日程度）
    は再利用する

────────────────────────────────────────
4-8. ゴール判定・ライド終了
────────────────────────────────────────

走行距離がルート総距離に到達した時点でゴールとみなし、「ゴール！」を
表示したのちライドを自動終了する。手動終了（終了ボタン）も可能。
終了時、記録サンプルが一定数以上あれば「ライド終了モーダル」（6-3節）を
表示する。

5. ワークアウトタブ
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

────────────────────────────────────────
5-1. 概要
────────────────────────────────────────

GPXルートを使わず、時間×パワー目標（FTPに対する比率）で構成される
インターバルトレーニングを実行するモード。標準的なZWO（Zwift Workout）
形式のXMLファイルをそのまま読み込める。

────────────────────────────────────────
5-2. ZWOワークアウトの読み込みと管理
────────────────────────────────────────

  【読み込み手段】
    - ローカルファイル: 端末上のZWOファイルを選択して読み込む（認証不要）
    - リモート: あらかじめ登録済みのZWOファイル一覧（表示名・合計時間付き）
      から選択してダウンロードする（GPXと同じSupabaseストレージを共用。
      DB仕様はdb.v4.md参照）。**招待ユーザーとしてログイン済みの場合のみ
      利用可能**（9-5節参照）。未ログイン時はボタンが無効化される
    - いずれの手段でも、読み込んだZWOファイルは名前・区間構成を解析して
      即座に保存する（GPXと異なり、名前の確認・編集ダイアログは挟まない）

  - 保存済みワークアウト一覧に、名前・合計時間・プロファイル概形
    （小型グラフ、区間ごとにFTP比率で色分け）を表示する
  - 選択すると、詳細プロファイル（大型グラフ）とワークアウト開始ボタンが
    有効になる
  - 削除可能（確認ダイアログあり）

  対応する区間種別:
    - ウォームアップ／クールダウン: パワーが直線的に変化する区間
    - 定常走行: 一定パワーを維持する区間
    - インターバル: ON/OFFパワーを指定回数繰り返す区間（内部的には
      定常走行区間の連続に展開される）
    - フリーライド: パワー目標を設けない自由走行区間（ERGモード解除）

  各区間は、FTPに対する比率でパワー目標を指定する（例: FTPの75%）。
  区間ごとにケイデンス目標（任意）を指定できる。

────────────────────────────────────────
5-3. ワークアウト実行画面
────────────────────────────────────────

  - 左側HUD: 実測パワー／目標パワー、実測ケイデンス／目標ケイデンス、
    心拍、経過時間・残り時間、仮想距離・仮想速度（参考値、勾配ゼロの
    平坦シミュレーションとして概算）
  - 右側: 区間一覧（現在区間をハイライト、残り時間を表示、自動スクロール）
  - 下部: 全体プログレスバー（区間ごとに色分けした帯グラフ、現在位置
    マーカー、総時間表示）
  - FTP微調整UI: ワークアウト実行中でもFTP値を上下ボタンで一時的に
    調整でき、目標パワーに即座に反映される
  - 目標からの乖離が大きい場合はアラートメッセージを表示する
  - 一時停止／終了ボタン

────────────────────────────────────────
5-4. ERGモード（自動負荷制御）
────────────────────────────────────────

対応トレーナーに対し、現在区間の目標パワーを一定間隔で送信し、
ケイデンスによらず目標パワーを維持させる（ERGモード）。フリーライド
区間ではERGモードを解除し、通常の自走状態に戻す。

────────────────────────────────────────
5-5. 自動一時停止・自動再開
────────────────────────────────────────

  - パワーがしばらくゼロの状態が続く、またはトレーナーが切断された場合、
    自動的に一時停止する
  - 一時停止中に一定以上のパワーが継続して検出されると、自動的に再開する
  - 手動で一時停止した場合も、いったんゼロパワーを経てから高パワーへの
    復帰が検出されると自動的に再開扱いとする（意図せぬ一時停止の解消）

────────────────────────────────────────
5-6. ワークアウト終了
────────────────────────────────────────

全区間終了、または手動終了で記録を確定し、終了モーダル（6-3節相当）を
表示する。位置情報・標高・勾配は記録しない（ルート非依存のため）。


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
6. ライド記録・エクスポート・Strava連携
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

────────────────────────────────────────
6-1. 記録サンプリング
────────────────────────────────────────

走行中（ルートモード・ワークアウトモードとも）、1秒間隔で以下を記録する:
  - タイムスタンプ
  - 位置（緯度・経度、標高）※ルートモードのみ
  - 走行距離、速度
  - 勾配 ※ルートモードのみ
  - パワー、ケイデンス、心拍

────────────────────────────────────────
6-2. ライド記録の保存
────────────────────────────────────────

記録は、走行終了時にユーザーが「保存」を選んだ場合にのみローカルに
永続化される（自動保存ではない）。保存すると以下の集計値も記録される:
  - 総距離、総経過時間
  - 平均パワー、平均ケイデンス（>0の値のみで平均）、平均心拍（>0の
    値のみで平均）
  - Strava アップロード状態（未送信／送信済／失敗）

────────────────────────────────────────
6-3. ライド終了モーダル／ワークアウト終了モーダル
────────────────────────────────────────

走行終了時に表示。サマリー（距離・経過時間・平均パワー・平均心拍）を
表示し、ライド名を編集できる（既定名は「gpx-vcycl route : {ルート名}」
またはワークアウト名ベース）。

  ボタン:
    [Stravaにアップロード]（Strava未接続時は無効化しラベルで案内）:
      1. 記録を保存
      2. FIT形式に変換
      3. Stravaへアップロード（処理完了までポーリング待機）
      4. 成功: アップロード状態を「送信済」に更新
         失敗: 「送信失敗」の旨を表示（記録はローカルに保存済みのまま）
    [保存のみ（後でアップロード）]:
      記録を保存するのみ（FIT変換・アップロードは行わない）
    [破棄]:
      何も保存せずに閉じる

────────────────────────────────────────
6-4. FIT形式へのエクスポート
────────────────────────────────────────

保存済みの記録は、いつでもFIT形式ファイルとしてダウンロードできる
（履歴タブから）。FITファイルには以下の情報が含まれる:
  - ファイルID、セッション（種目=サイクリング、サブ種目=バーチャル走行
    またはインドアサイクリング）、ラップ、レコード（1秒毎のサンプル）
  - GPS位置・標高（ルートモードのみ）、速度、パワー、ケイデンス、心拍
  - 平均値・獲得標高（ルートモードのみ）などのサマリー

────────────────────────────────────────
6-5. Strava連携
────────────────────────────────────────

  - 設定モーダルからStravaアカウントと連携（OAuth認可、外部ブラウザ
    遷移を伴うためBLE接続は一旦切断される。ライド開始前に接続しておく
    ことを推奨）
  - 連携解除も可能
  - Stravaアップロード時、ワークアウト記録は「屋内トレーニング」として、
    ルート記録は「バーチャルライド」としてアップロードされる
  - ライド開始前にStrava未接続の場合、確認ダイアログを表示し、未接続の
    まま続行するか選択させる（ダミートレーナー使用時はこの確認をスキップ）

────────────────────────────────────────
6-6. 履歴タブ
────────────────────────────────────────

保存済みライドを新しい順に一覧表示する。各記録について:
  - ルート名、Stravaアップロード状態バッジ、日時、距離、経過時間、
    平均パワー
  - Strava送信済みの場合、Strava上の該当アクティビティへのリンク
  - 操作: 再アップロード（未送信または失敗時）、FITダウンロード、削除
    （確認ダイアログあり）

8. データモデル（ローカル保存領域）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

端末ローカルの永続ストレージに、以下の論理データを保持する
（実装形式は implement.txt 参照。リモートGPXソースのスキーマは
db.v4.md 参照、本書には含めない）。

────────────────────────────────────────
8-1. ルート
────────────────────────────────────────
  - ルート名、GPX原文、総距離、総獲得標高、保存日時

────────────────────────────────────────
8-2. ライド記録
────────────────────────────────────────
  - 紐づくルートID・ルート名（ワークアウト記録の場合は null／
    ワークアウト名）、開始・終了日時、総距離、総経過時間、平均パワー・
    ケイデンス・心拍、1秒毎のサンプル列、Strava連携状態
    （未送信／送信済／失敗、送信日時、StravaアクティビティID、
    失敗時のエラーメッセージ）

────────────────────────────────────────
8-3. ワークアウト
────────────────────────────────────────

────────────────────────────────────────
8-4. 設定
────────────────────────────────────────

  - Strava連携トークン（アクセストークン、リフレッシュトークン、
    有効期限、アスリートID・名前）
  - 各BLEデバイスの直近接続デバイス識別子（再接続補助用）

────────────────────────────────────────
8-5. 実写風景写真キャッシュ
────────────────────────────────────────
  - ルート×地点ごとに選定済みの画像情報（一定期間キャッシュ）

9. 外部サービス
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

────────────────────────────────────────
9-1. Strava
────────────────────────────────────────
  用途: ライド記録のアップロード（OAuth連携）
  認可コードのアクセストークンへの交換、およびトークンのリフレッシュは、
  クライアントの秘密情報を含むため、サーバー側（本体アプリとは別の
  軽量なプロキシ処理）を経由する。

────────────────────────────────────────
9-2. 実写風景写真データベース
────────────────────────────────────────
  用途: 4-7節の風景写真取得。アクセストークン未設定時は機能自体を
  無効化する（フォールバックなしで非表示）。

────────────────────────────────────────
9-3. リモートGPX/ZWOソース
────────────────────────────────────────
  用途: 4-1節のリモートGPX読み込み、5-2節のリモートZWO読み込み。
  同一のSupabaseストレージ・プロジェクトを共用し、ファイル種別は
  拡張子（.gpx/.zwo）とメタデータテーブル（route_files/workout_files）
  で区別する。スキーマは db.v4.md 参照。ワークアウトのアップロードは
  本アプリの対象外（別ツールが担当。db.v4.md参照）。

────────────────────────────────────────
9-4. 地図・ストリートビュー（オーナーモード限定）
────────────────────────────────────────
  用途: 開発者本人のみが使う代替地図プロバイダ。一般利用時は使用しない
  （7章の地図プロバイダ設定はオーナーモードでのみ表示される）。

  3Dマップ（走行風景）は地図プロバイダによらず常時表示する。
  Googleモード時は、右カラムの写真枠をStreet View、地図枠をGoogle Maps
  として使う（Mapillaryとは排他。実写風景写真機能・4-7節は無効化される）。
  レイアウト比率・切り替え挙動はOSMモード（Mapillary/Leaflet）と同一。

────────────────────────────────────────
9-5. 招待制ログイン（Supabase Auth・マジックリンク）
────────────────────────────────────────
  用途: 9-3節のリモートGPXソースへのアクセス制御。gpx-navi・gpx-editor
  （同じSupabaseプロジェクトを共有する姉妹アプリ）と共通の認証方式を
  踏襲する。

  【方式】
    - パスワード無しのマジックリンク認証。サインアップ機能はなく、
      あらかじめSupabase側で登録済みの招待ユーザーのメールアドレス
      でのみログインできる
    - ヘッダー右上の歯車アイコン（⚙️）をクリックするとログイン用
      ポップオーバーが開く
      - 未ログイン: メールアドレス入力欄＋「ログインリンクを送る」
        ボタン＋「キャンセル」ボタン。送信するとメールに届いた
        リンクをクリックすることでログインが完了する（ブラウザに
        戻ってきた時点で自動的にセッションが復元される）
      - ログイン中: 「ログイン中: {メールアドレス}」表示＋
        「ログアウト」ボタン＋「キャンセル」ボタン
      - どちらの状態でも「キャンセル」ボタンは状態を変更せず
        ポップオーバーを閉じるだけ
    - ログイン状態はブラウザに保持され、再訪問時も維持される

  【ゲーティング範囲】
    - リモートGPX読み込み（4-1節）・リモートZWO読み込み（5-2節）の
      2つのみがログイン必須。未ログイン時は「ネットワークからGPXを
      読み込む」「ネットワークからZWOを読み込む」ボタンがそれぞれ
      無効化される
    - それ以外の全機能（ローカルGPX/ZWO読み込み、ライド、ワークアウト、
      BLE接続、Strava連携等）はログイン不要で利用できる
    - このゲーティングはUI上の制御であり、実際のアクセス制御は
      Supabase側のRow Level Security（RLS）ポリシーで行う

10. エラーハンドリング・異常系一覧
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  場面                             対応
  ──────────────────────────────   ─────────────────────────────────────────────
  BLEデバイスが見つからない        ログにエラー表示、ボタンを「接続」に戻す
  ライド中にBLE切断                ログに切断表示、直前値を保持したまま更新停止
  トレーナー制御ポイント非対応     ログに「制御ポイント利用不可」、負荷制御なしで
                                    読み取りのみ継続
  トレーナー命令タイムアウト(3秒)  エラーをログ、次回送信タイミングで再試行
  急勾配（±25%超）のGPX           ±25%にクリップして送信（トレーナー保護）
  elevationタグなしGPX             勾配0%として扱う（エラーにしない）
  GPXポイント間隔が広い区間        隣接点間の傾斜にフォールバックして計算
  不正なGPXファイル                パースエラーを表示し読み込みをスキップ
  有効なトラックポイントが2点未満  エラーとして読み込み中断
  ワークアウト実行中の自動一時停止 5-5節参照
  ライド開始時Strava未接続         確認ダイアログ、続行するか選択可能
                                    （ダミートレーナー時はスキップ）
  Strava認可コード交換失敗         トースト通知でエラーを表示
  Stravaアップロード失敗           ローカル保存は維持したままエラー表示、
                                    履歴タブから再アップロード可能
  Stravaアップロード処理タイムアウト（約60秒） エラーとして扱う
  風景写真の候補が見つからない／
  角度が大きくずれる                「No Image」プレースホルダーを表示

11. 非機能要件
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  対象デバイス:
    - iPadOS 16+ ＋ Bluefyブラウザ（Web Bluetooth利用のため）
    - 開発・デバッグ用にデスクトップChrome/Edgeでも動作すること
      （BLE部分はモックモードで代替）

  通信要件:
    - HTTPS必須（Web Bluetoothの要件）。ローカル開発時も自己署名証明書
      等でHTTPS化する

  画面:
    - ランドスケープ（横向き）を前提とする
    - ライド中・ワークアウト中は画面の自動スリープを抑止する
      （Wake Lock）

  オフライン対応:
    - PWAとしてインストール可能（ホーム画面追加）
    - 静的アセットはService Workerでキャッシュする
    - GPXルート・ライド記録・設定はローカル永続ストレージに保存し、
      オフラインでも閲覧・走行が可能（新規のリモートGPX読み込みや
      Stravaアップロードはオンライン必須）

  単位・表記:
    - UIラベル・メッセージは日本語
    - 内部の物理量はSI単位を基本とし、表示時のみ変換する
      （速度: km/h表示、内部m/s、等）

  スコープ外（明示的にやらないこと）:
    - マルチユーザー対応・ソーシャル機能
    - アバター・3Dグラフィック演出（Zwift風のキャラクター表示等）。
      マップ・走行風景表示に限定
    - ルート・記録のクラウド同期（ローカル保存が基本。Stravaアップロード
      は片方向の送信のみ）
    - ANT+対応（BLEのみ）
    - 複数トレーナーの同時接続（1台のみ）

12. バージョン管理
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

アプリのバージョンはビルド時定数としてアプリ本体に埋め込む。

  バージョン形式: セマンティックバージョニング (X.Y.Z)
    パッチ (x.y.Z): バグ修正・細かい調整
    マイナー (x.Y.0): 新機能追加・UI変更
    メジャー (X.0.0): 大規模リファクタ・破壊的変更

  表示場所: ヘッダーのアプリ名の右隣に「vX.Y.Z」形式で、
    アプリ名より小さく控えめな色で表示する。
